# FinWatch — Limitations and future work

This document is the consolidated, defensible statement of what the FinWatch thesis project does **not** do. Mentioning a limitation here is intentional — the alternative ("we never thought about it") is the answer that loses thesis defense points.

The committee should expect every item here to be revisited as future work, with a path described — not as an open hole.

---

## 1. Data

### L1.1 Synthetic data only
All transactions, accounts, and merchants in the demo are generated by `finwatch/scripts/generate_transactions.py`, `simulate_fraud.py`, and the baseline seed in `prepare_demo_full.py`. No real bank or customer data was used; no IRB / ethics review was required.

**Implication for results:** precision/recall/F1 numbers in `finwatch/evaluation/report_*.md` are bounded by the realism of the synthetic distribution. They are useful as relative measurements (rule A vs rule B), as a floor (the rule clearly detects the patterns it was designed for), and as a reproducibility benchmark (same seed → same numbers). They are not transferable to production environments unchanged.

**Path:** the architecture supports straightforward replacement of the synthetic generator with a real ingestion connector against a sanitized production dump. The downstream pipeline + rules do not need to change.

### L1.2 Amount distribution
`generate_transactions.py` draws transaction amounts from a log-normal distribution per transaction type (calibrated in mid-2026 to roughly match retail / B2B mix in VND). Real-world distributions are merchant-specific, time-of-day specific, and have heavier tails than log-normal in the upper percentiles.

**Path:** fit a Pareto-mixture model to a 30-day production sample once available.

### L1.3 No temporal patterns
Synthetic traffic is uniform throughout the day. Real card transactions cluster heavily around lunch + early evening; salary transfers cluster mid-month; e-commerce spikes on weekends. The VELOCITY and FAIL_SPIKE rules use rolling windows that are insensitive to these patterns, but a more sophisticated detection layer (e.g. seasonal-adjusted z-score) would need time-of-day modelling.

### L1.4 No merchant-level realism
Merchants in the seed data are 12 hand-crafted names with hand-assigned `risk_level`. Real merchant networks have thousands of entries with mature risk scoring tied to chargeback history, sanctions lists, and merchant category codes (MCC).

## 2. Detection

### L2.1 Rule-based, no ML scorer
The 6 anomaly rules in `finwatch/clickhouse/queries/anomaly_*.sql` are explicit, explainable SQL aggregations with manually-set thresholds. There is no machine-learning component (no gradient boosting, no isolation forest, no neural model).

**Rationale (kept intentional):** rule-based detection is explainable — an analyst opening a case in `/alerts` can see exactly why the rule fired. ML scorers improve recall but trade away explainability and require labelled training data the thesis does not have.

**Path:** ADR-0006 sketches the overlay design — a secondary ML scorer consuming the same Kafka topic, writing `ml_score` to `fraud_alerts`, used as a re-ranker not a primary signal.

### L2.2 Thresholds not optimized
The thresholds in each anomaly rule are documented and justified in each SQL file's header (see audit item C2 commit), but no formal tuning loop optimized them. The evaluation in `finwatch/evaluation/report_*.md` reports F1 at the shipped thresholds; no grid search or Bayesian optimization was performed.

**Path:** once labelled production data is available, run a per-rule threshold sweep maximizing F1 (or precision at fixed recall, depending on the cost of false positives in the operating context).

### L2.3 No per-merchant or per-counterparty analysis
Detection is account-centric. A merchant that legitimately moves 250M+ VND per transaction (e.g. car dealership) will produce LARGE_AMT alerts on every payment. There is no merchant whitelist, no per-counterparty norm.

**Path:** add a merchant baseline table parallel to the account baseline, then a relative-to-merchant z-score rule.

### L2.4 No graph signals
Mule rings, layering chains, and smurfing all manifest as graph patterns (one account funds many others, money flows through a chain). The pipeline materializes per-account aggregates only — there is no graph view.

**Path:** out of scope. A graph layer (Neo4j or a CH `Distributed`-backed adjacency table) would be a separate sub-project.

## 3. Compliance and governance

### L3.1 No SAR / OFAC / regulator output
The system does not generate Suspicious Activity Reports in the form required by SBV Circular 35/2018 or by foreign regulators (FinCEN SAR, EU SDD reports). No OFAC / EU sanctions list screening is performed at transaction time.

**Path:** the existing `fraud_alerts` table has the audit-trail skeleton (`evidence` JSONB, `status`, `notes`, `created_at`). Adding a SAR generator is a downstream consumer that reads `fraud_alerts` and renders a regulator-formatted report — does not require pipeline changes.

### L3.2 No structuring detection
A 1B VND transfer split into eleven 90M transfers will evade LARGE_AMT entirely while passing under HIGH_RISK if the destination merchant is low-risk. The current VELOCITY total-amount clause catches some splits but not those spread across 6+ minutes.

**Path:** dedicated structuring rule using (account, time-bucket) aggregates over 1-hour and 1-day windows; called out under H4 in the audit.

### L3.3 No audit log on UI actions
The `/accounts/[id]` Suspend / Reactivate buttons hit `POST /api/accounts/{id}/lock` and `/unlock`, but neither records who clicked (no authentication) nor when. For real fraud-ops, every analyst action needs an immutable audit trail with operator identity.

**Path:** add an `action_log` table with FK to `accounts` and `analyst`; populate from the API route once auth is wired (item L4.1).

### L3.4 Case workflow incomplete
The `/alerts` page lists cases but does not let an analyst close them. Cases stay in status `open` forever. In production, an analyst would mark each case as `closed_fraud`, `closed_clean`, `escalated`, or `merged`.

**Path:** add `PATCH /api/alerts/:id` with state-machine transitions; UI buttons for the four states; reflect in the audit log (L3.3).

## 4. Production-readiness

### L4.1 No authentication
The web UI at `http://localhost:3002` is unauthenticated. Any client on the network can `POST /api/accounts/:id/lock` or insert transactions. Intentional for the demo on a localhost stack.

**Path:** NextAuth middleware at `web/middleware.ts`; OAuth proxy in front for production deployment. The architecture does not need to change to add this.

### L4.2 No TLS between services
PostgreSQL ↔ Debezium, Debezium ↔ Kafka, Kafka ↔ ClickHouse — all use plaintext TCP/HTTP. Intentional for a single-host Docker Compose demo.

**Path:** Kafka KRaft + SASL_SSL listeners; Debezium PG client certs; ClickHouse mutual TLS — all are documented Docker Compose configurations, just not enabled here.

### L4.3 No high-availability topology
The compose file runs a single Kafka broker, a single PG instance (no replica), and a single ClickHouse node. Broker crash = pipeline halt.

**Path:** production topology requires:
- 3-broker Kafka cluster with `replication.factor=3` for the CDC topics
- PostgreSQL streaming replica (`max_replication_slots=10`)
- ClickHouse 2-shard / 2-replica cluster sharded by `account_id` hash

RPO/RTO targets in this single-host topology: RPO = last commit (no data loss with PG durable WAL), RTO = container restart time (~10–30 s). Production targets: RPO ≤ 1 min (sync replica), RTO ≤ 1 min (broker failover).

### L4.4 No dead-letter queue
The ClickHouse Kafka engines use `kafka_skip_broken_messages=10` to tolerate Debezium schema drift, but malformed messages are silently discarded — there is no DLQ topic to inspect them.

**Path:** add a `finwatch.dead_letter` topic and a small consumer that writes poison messages to a queryable table.

### L4.5 Observability is partial
`prometheus/prometheus.yml` scrapes ClickHouse on port 9363 only. Kafka, Debezium, PostgreSQL, fraud-worker, and the web app expose no metrics. Logs are unstructured stdout. No distributed tracing.

**Path:**
- Easiest win: `prometheus_client` in `fraud_alert_worker.py` exporting `fraud_alerts_emitted_total{rule}` and `worker_tick_duration_seconds`.
- Medium effort: Kafka JMX exporter, Debezium connector metrics endpoint.
- Larger: OpenTelemetry instrumentation across the Next.js + Python boundary for correlation-ID propagation.

### L4.6 Hardcoded credentials in `.env.example`
Default passwords `finwatch_secret_2024` and `clickhouse_secret_2024` ship in the repo. These are documented as **local-Docker-only** defaults (see the comment header in `.env.example`). They are inappropriate for any non-localhost deployment.

**Path:** secrets management via HashiCorp Vault or AWS Secrets Manager; the inline fallback defaults have been removed from the Python scripts so they now fail-fast at startup if the env var is missing (audit H2).

## 5. Evaluation methodology

### L5.1 Per-account, not per-transaction
The evaluation in `finwatch/evaluation/report_*.md` measures whether a rule flags the *correct accounts*. Per-transaction precision/recall would be a separate evaluation — it matters for analyst workload (how many transactions does an analyst need to inspect per case?) but does not change the rule's decision boundary.

### L5.2 ZSCORE evaluated on synthetic baseline
Each ZSCORE victim has only 20 backdated baseline transactions in the evaluation. Production accounts have hundreds. The reported ZSCORE F1 is a floor; with a longer baseline, the stddev sharpens and recall typically improves.

### L5.3 Reproducibility limited to seed
The eval script captures `git rev-parse --short HEAD` in the report header but does NOT capture Docker image digests or ClickHouse `stream_flush_interval_ms`. A report generated against a different ClickHouse patch version could legitimately differ on edge cases.

**Path:** extend `collect_evidence.py` (and reference in the eval report header) to dump `docker compose images --digests` and the active ClickHouse profile settings.

## 6. Scope boundary

To be explicit about what this thesis is **not** trying to be:

- It is not a production fraud-detection platform.
- It is not a comparative study of streaming engines (we picked one architecture and went deep; Spark Streaming / Flink would be separate theses).
- It is not a machine-learning thesis (rule-based by deliberate choice — see ADR-0006).
- It is not a Vietnamese banking compliance thesis (SAR / OFAC out of scope per L3.1).
- It is not a high-availability deployment thesis (single-host Docker Compose per L4.3).

What it **is**: a reference implementation showing that sub-second end-to-end CDC + rule-based fraud detection is achievable on commodity hardware (1 Docker host, 16 GB RAM, sustained ~1500 TPS) — with reproducible benchmarks and per-rule evaluation against labelled synthetic data.

See [`docs/contribution.md`](./contribution.md) for the explicit contribution claim and [`docs/related-work.md`](./related-work.md) for the comparative architectural analysis.
